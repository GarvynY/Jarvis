## 项目简介

该项目围绕澳大利亚大选相关的公开社交媒体数据展开，覆盖 Mastodon 与 Bluesky 两个数据来源，并对选举相关内容进行采集、清洗、情绪分析、地理位置映射和时间趋势分析。系统最终通过交互式分析看板展示不同城市、时间维度和关键词下的公众情绪变化。

项目部署在 Nectar Cloud 的 Kubernetes 集群上，核心组件包括 Fission 无服务器函数、Redis + RQ 异步任务队列、Elasticsearch 三节点集群和 Jupyter 可视化分析环境。系统覆盖 9 个澳大利亚主要城市，支持多月历史数据采集与增量更新，并构建了热力图、词云、时间趋势等多种可视化分析模块。

## 系统架构

系统采用"定时触发 + 异步执行 + 两阶段处理 + 搜索索引 + 可视化分析"的整体架构。

Fission 函数按小时触发数据采集任务，但不直接执行长时间采集逻辑，而是将任务写入 Redis 队列。后台 RQ Worker 负责执行实际的数据采集、清洗、情绪分析、地理位置增强和聚合处理。处理结果写入 Elasticsearch，用于后续全文检索、地理查询、多维聚合和可视化分析。

这种设计将任务调度与任务执行解耦，避免了无服务器函数在长时间 API 调用和分页采集场景下超时，也提升了系统的稳定性、可恢复性和扩展能力。

## 数据处理流程

系统的数据处理流程分为两个阶段：

**第一阶段：数据采集与基础处理**

该阶段负责从 Mastodon 和 Bluesky 采集选举相关公开数据，并进行基础清洗和增强处理，包括文本清洗、情绪分数计算、地理位置推断和时间字段标准化。

**第二阶段：聚合与分析加工**

该阶段在原始数据基础上进行多维聚合，例如按城市、时间、情绪类别和关键词进行统计，并将结果写入 Elasticsearch，供后续可视化分析使用。

两阶段架构的优势在于，原始数据与分析结果相互解耦。当分析逻辑发生变化时，可以在不重新采集数据的情况下对历史数据进行重算，从而提升系统的可维护性和迭代效率。

## 技术亮点

### 无服务器定时调度

系统使用 Fission 作为无服务器运行时，通过定时触发器自动启动采集任务。由于社交媒体采集是周期性任务，而非持续运行服务，因此无服务器调度可以降低长期资源占用，并保持数据的持续更新。

### 异步队列解耦长任务

社交媒体 API 采集往往涉及分页请求、限流和历史数据回溯，执行时间较长。系统通过 Redis + RQ 将任务调度与任务执行分离：Fission 函数只负责派发任务，后台 Worker 负责执行耗时的数据采集与处理，从而避免触发器函数超时，并提高任务恢复能力。

### 两阶段处理架构

系统将数据采集与数据分析拆分为 Harvest 和 Refine 两个阶段，使原始数据保留、历史数据重算和分析逻辑迭代更加灵活。该设计也降低了数据管道不同模块之间的耦合度。

### 搜索索引与可视化分析联动

系统使用 Elasticsearch 存储原始数据和聚合结果，支持全文搜索、地理空间查询和多维聚合。前端通过 Jupyter、Folium、时间序列图和词云等形式展示分析结果，实现从数据采集到分析展示的闭环。
